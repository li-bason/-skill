# StudyForge

把散落的学习资料、真实问题和持续练习，锻造成可追溯、可更新、可验证的学习资产。期末复习是高频入口，
底层能力可服务考研专业课、职业资格认证、企业培训和课程助教；个人降低整理成本，教师获得可审计的覆盖与
题库草案，机构沉淀标准化、可复用的课程知识资产。

将本文件所在目录记为 `SKILL_DIR`。所有资源路径均相对于 `SKILL_DIR` 解析。

## 强制全流程合同

**本 skill 一旦被触发，以下全部流程立即成为强制执行合同，而不是可选建议。**

1. 必须从“输入归一化”开始，依次完成任务设计、用户确认、合同锁定、正式产出、质量校验、
   会话/进度更新和 gate 检查；不得跳过中间阶段后直接交付。
2. 必须读取本 `SKILL.md` 所路由的工作流、质量检查和来源规则，并执行其中全部适用条款；
   不得只依据用户最新一句反馈做局部修补。
3. 用户指出一个例子时，必须把它视为同类问题的测试样例，扫描并修复全部产物中的同类问题，
   再重新执行全流程校验。
4. 脚本或校验器通过不等于任务完成。还必须人工审查内容相关性、可读性、公式、来源、格式以及
   `review_spec.md`、`review_lock.yaml`、`progress-log.md` 与正式产物的一致性。
5. 只有用户明确要求“只做某一步、只修改某个文件或只修这个例子”时，才允许部分执行；
   必须在交付中说明未执行的阶段和未校验范围。除此之外一律按完整流程执行。
6. 若任一强制阶段未完成、任何 GATE 未满足、合同与产物不同步或质量检查仍有 error，
   禁止声称“已完成”“已交付”或“校验通过”。
7. 当前 `demo/数据挖掘` 是输出结构的黄金回归样例。任何模板、生成器或校验器修改后必须运行
   `scripts/test_golden_structure.py`；黄金样例或任一模板结构测试失败时禁止打包交付。

## 执行纪律

1. 严格执行：输入归一化 → 任务设计 → 用户确认 → 合同锁定 → 正式产出 → 校验交付。
2. 进入每个阶段前检查 GATE；标记为 BLOCKING 时停止并等待用户回答。
3. 每次只问一个问题。已有材料能够推断时先给推荐值，不重复询问。
4. 在用户确认任务设计前，不批量生成正式复习正文。
5. 每完成一个阶段，更新任务目录中的 `progress-log.md`。
6. 不声称读取了未成功解析的材料，不编造教材覆盖、原题来源或引用页码。
7. Markdown 是规范主产物；用户需要时同步生成 PDF。
8. `tmp/`、`tmp_pdf_check/` 等临时目录不得预先创建或作为正式产物保留。只有在 PDF 渲染、OCR、
   截图或视觉验收确有需要时才按需创建；验收完成后删除其中的中间文件和空目录。临时目录
   永远排除在 Skill 打包、正式交付和常规内容扫描之外。
9. 用户明确给出课程名、考试大纲或题型分布时，它们是最高优先级；不得用不同课程的文件标题、目录或题型分布替代。
9. 大纲条目不是天然的最细考点。先拆分为可独立定义、比较、流程说明或计算的**原子考点**；不得把“概念 + 实现步骤 + 计算过程”等并列要求压成一个节点。
10. 用户要求保留来源内容时，`知识点.md` 必须逐原子考点摘录能够**直接回答该考点**的来源原句，并保持这些句子的文字完整。不得整页照搬，也不得把背景故事、目录、重复说明、无关案例或同页其他知识混入该考点。
11. 上述规则必须覆盖**全部大纲条目**，不是只处理用户举例的条目。建立大纲条目到原子考点的一对多映射；任何大纲条目为零原子点、任何原子点缺少来源原文、任何原子点少于 3 题，都禁止交付。语言积累路线不生成题库，不适用最低题量规则。
12. “几个重要概念”必须继续拆到来源中实际列出的每个概念；题量按最终原子点数计算。例如来源列出 `n` 个概念，另有 1 个算法流程，则至少生成 `(n+1)×3` 题，不能按上级大纲的一句话只生成 3 题。
13. 同一条目或同一页中列出多个不同方法、算法、指标或任务时，每一个名称必须单独建立原子考点。不得把“智能推荐、时间序列、度量学习”这类不同主题放入同一个知识点，也不得将它们的来源文字拼成一段。
14. 同一原子考点包含“目标、定义、区别、步骤、优点、缺点、参数”等多个信息项时，每项必须独立成段，段间留空行。允许删除 `Ø`、`•` 等 PPT 项目符号和“与分类区别”中的冗余连接词，但不得把两个信息项压在同一段。
15. PDF 文本中的 `??`、错位上下标、零宽字符、私用区数学符号或无法辨认的公式片段一律视为解析失败。回看原 PDF 页面或图片，人工还原为清晰的 Unicode/Markdown 数学表达，并标注来源页；无法确认时删除该片段并标记 `unverified`，绝不能把乱码写入正式资料。
16. 材料存在知识缺口、页面缺失、公式无法从视觉页面确认，或来源对概念解释不足时，先复核用户材料和原始页面，再按缺口规模自动分流。只有缺口局部且数量很少（默认同时满足：不超过 3 个原子考点、且不超过全部考点的 10%）时才检索少量权威 Web 来源；技术内容优先原始论文、官方文档和教材出版社页面。缺口超过任一阈值时禁止逐站检索，直接使用模型知识补齐必要解释并标记为 `inferred`，不得伪造 URL、页码或教材原文。进入该模式时必须在正式生成前告知用户：补充内容缺少完整资料支撑，质量可能不符合预期，需要仔细核对知识点；交付摘要中再次保留同一风险提示。联网内容登记为 `external`，记录 URL、标题、访问日期和它补充的原子考点。
17. Web 只补充缺失或无法核验的部分，不覆盖教师大纲和用户材料中已经明确的考试口径。若材料与权威外部来源表述不同，保留双方并记录冲突，复习口径默认以用户大纲和教师材料为准。
18. `知识点.md` 每点只保留“原子点标题 → 考查要求 → 分块内容”。定义、判定条件、公式、例题、推导和结论分别成块；公式独占一行，连续推导使用有序步骤。禁止重复编号表、来源表、空标题和大段表格。图中坐标、节点取值、箭头标签、页码、水印、题型目录或示意数据不得进入正式内容。
19. 文科、理科、工科共享同一内容契约：章节目录、题型说明、页眉和同页其他考点不得进入正文；
    每点使用“考查要求＋分块内容”；去除重复摘录；题目只考当前原子点；Markdown 通过后仍须
    渲染目检。学科差异只体现在分块：文科使用概念/背景/原因/影响/论据/评价/结论，理科增加
    定义/定理/公式/推导/例题/结论，工科增加原理/参数/流程/伪代码/应用/注意事项/结论。
20. 所有板块统一执行逐点排版：无先后顺序的并列要点使用项目列表，每点独占一行；有先后
    顺序的步骤、推导、论证、流程和解题过程使用编号列表，每步独占一行。禁止在一行中连续
    塞入 `①②③`、`第一…第二…第三…`、多个项目符号或多个步骤；禁止把分点内容塞进表格单元格。
21. 语言分类只执行积累与增量整理，维护 `词汇语法积累表.md`、`翻译积累表.md`、
    `写作积累表.md` 三个独立表格；不创建阅读板块、题目、答案或思维导图。单词、词组、翻译
    或写作句子必须先结构化展示，再询问是否收录；用户确认前禁止写入。写作表只积累英文句子。

## 路由

- 新建期末、考研专业课、职业资格认证或其他课程型学习任务：读取 `workflows/final-exam.md`。
- 新建答题型竞赛备赛：读取 `workflows/contest-review.md`。
- 用户说“继续、恢复、接着做、更新、补充新材料”：读取 `workflows/resume-review.md`。
- 交论文、提交作品或实操型竞赛不走竞赛刷题流程；向用户说明边界。
- 企业培训或课程助教任务复用课程型工作流，将“考试范围”解释为培训目标、课程目标或能力清单，将题库作为可选产物。
- 用户目标不清时，根据现有材料选择最匹配路线；只有不同路线会实质改变产物且无法推断时，才询问任务目标。

## 通用流水线

### Step 1：输入归一化

**GATE**：用户提供了材料、正文内容，或明确说明了复习目标。

使用当前环境最合适的文档、PDF、表格、图片/OCR能力读取材料，再用
`scripts/import_sources.py` 将文件和解析状态登记到任务的 `sources/`，生成
`source_manifest.json`。脚本可直接解析常见文本格式，并在相关 Python 库可用时解析
PDF、DOCX、PPTX、XLSX；ZIP 仅安全解压后逐项登记。

**对话文本同样是来源文件。** 用户在消息中直接给出的考试大纲、题型/分值、教材目录、
题目、答案、笔记、修订意见或范围声明，必须在解读前原样写入临时 UTF-8 文件，再登记为
`sources/SRC-xxx-<title>.md`：

```bash
python scripts/register_text.py "<任务目录>" --title "用户提供的复习大纲" --input "<临时原文文件>" --role primary --delete-input
```

不得只在 `review_spec.md` 中转述对话文本，也不得把对话来源写成不存在的页码。后续正式
产物、范围核对表和合同应引用该 `SRC-xxx`。

解析失败不得阻塞其他材料，但必须把状态写成 `needs_conversion` 或 `needs_ocr`，并只向
用户追问最关键的一个缺口。

对每个 `needs_conversion`、`needs_ocr`、公式乱码或内容不足的原子点执行：

1. 回看原文件视觉页面或更换解析方式；
2. 仍无法得到可用内容时，使用 WebSearch/WebFetch 检索权威来源；
3. 将网页登记到 `source_manifest.json`，状态设为 `external`，记录 URL、标题和访问日期；
4. 在知识点中把“材料原文”和“外部补充”分开显示；
5. 仍无法确认则标记 `unverified`，禁止自行猜测。

先统计缺口覆盖的原子考点数和比例。缺口不超过 3 个且不超过全部考点的 10% 时执行第 2-4 步；超过任一阈值时自动跳过第 2-4 步并进入快速补全模式，无需等待用户明确要求。模型知识只承担解释、定义和例题补全，状态必须为 `inferred`；教师大纲、题型和用户 PDF 继续保持最高考试口径。进入快速补全模式时，生成前和交付摘要中都必须提示“补充内容缺少完整资料支撑，质量可能不符合预期，请仔细核对知识点”。

在生成任何复习正文前，建立“课程与范围核对表”：记录用户声明的课程名、每份材料的
标题/章节、与课程的关系（`primary`、`supporting`、`excluded`）及原因。若两份材料课程名
或章节主线不一致，必须停止把它们混合为同一考试范围；只把用户明确指定的大纲作为
`primary`，其余材料最多作为经标注的辅助背景，或标记为 `excluded`。

然后创建 `atomic_points.md`。为每个原子考点记录 `atomic_id`、上级大纲条目、考查动作
（定义/比较/流程/计算/应用）、来源 ID、来源页/段和题量要求。遇到“与、及、以及、分别、
步骤、计算过程、优劣、代表算法”等并列或可分开评分的表达，默认拆分；除非用户明确要求合并。
完成后逐条反查用户大纲：每个最上层编号及其每个并列子要求都必须至少映射一个
`atomic_id`。不得因为用户只指出某一处错误，就只修该处。

#### 考点先行、按需抽取（强制）

除考试大纲、题型分布、教师明确复习要点等范围文件外，教材、讲义和资料 PDF/PPTX 默认只登记文件与元数据，不得先整份转换为 Markdown。先解析范围文件并建立 `atomic_points.md` / `atomic_map.json`，再依据原子考点的标题、同义词和考查动作，对 supporting 材料执行页级检索，只抽取命中页及必要相邻页。未命中的页面不进入上下文，也不生成 Markdown。

```bash
python scripts/import_sources.py "<任务目录>" --metadata-only <教材或讲义...>
python scripts/extract_relevant_pdf.py "<资料.pdf>" "<任务目录>/atomic_map.json" "<任务目录>/sources/<SRC-ID>.md"
```

只有范围文件缺失、无法形成考点，或定向检索连续漏掉用户明确指定的内容时，才允许扩大抽取范围；扩大范围仍应分章节进行，不得直接把整本资料送入模型上下文。

### Step 2：任务设计

**GATE**：已知道任务类型和名称，且存在至少一项可用输入或用户给出的实质内容。

运行 `scripts/task_manager.py` 创建任务目录。读取
`workflows/template-selection.md`，选择真实模板文件。生成 `review_spec.md`，包含：

- 范围与目标。禁止询问考试日期或可用复习时间；它们不参与交互、GATE 或 BLOCKING。
  用户主动提供时可记录，但不得围绕这些信息追加问题
- 材料优先级与输入缺口
- 模板选择及理由
- 输出文件和内容深度
- 题型、难度比例与补题策略
- 课程与范围核对表，以及被排除材料的原因
- 来源追踪、联网补充与更新策略

### Step 3：设计确认

**BLOCKING**：用一个简洁推荐方案向用户确认范围、输出结构、题型和难度策略。默认难度
比例为基础 40%、中等 40%、综合 20%。用户确认后再生成 `review_lock.yaml`。

### Step 4：合同锁定与正式产出

**GATE**：用户已确认 `review_spec.md`。

从 `references/review_lock_schema.md` 生成严格 YAML/JSON 兼容合同，并执行：

```bash
python scripts/validate_contract.py "<任务目录>/review_lock.yaml"
```

按合同逐文件生成产物。每个知识点、题目或遗漏点使用稳定 ID，并记录来源状态：
`provided`、`external`、`inferred` 或 `unverified`。无法验证的内容不得伪装成教材内容。

文科、理科、工科正式产物分为两层：

- `知识点.md`：使用固定结构“原子点标题 → 考查要求 → 分块内容”。只放能直接回答该点的
  定义、公式、规则或步骤；章节目录和题型说明只用于识别范围，不进入正文。资料只有术语名称时
  不得视为覆盖完成，必须补全可学习内容并标记 `inferred`。
- `题目.md` / `答案.md`：作为训练题库；每个原子考点至少 3 题，并用 `atomic_id` 标注题目和答案。

`scripts/build_atomic_bank.py` 只能建立待审题库草稿，不能直接交付。每道选择题必须经过语义审查：
题干考查当前原子点，四个选项处于同一语义层级，干扰项对应真实易错点。章节标题、题型目录、
其他考点原句和“是否符合课程材料”不得用作题干或选项。

正式题库不得重复套用“完整说明、围绕辨析、结合场景”之类通用题干。每题必须包含能够独立
作答的具体对象、陈述、选项、空缺、数据或步骤要求；选择题必须有有效干扰项，计算题必须有
数值和计算条件，答案必须给出具体结论及必要过程。同一原子点的 3 题应分别考查识记、理解和
应用/计算，而不是换三个动词重复问同一件事。

选择题固定使用逐行格式：题干单独一行，随后 `A：`、`B：`、`C：`、`D：` 各占一行；
题干与 A、相邻两个选项之间必须保留 Markdown 空行，保证预览模式也分别成段；禁止只依靠
源码中的普通换行，更禁止把四个选项挤在同一行。

`答案.md` 不使用循环的 `1、2、3` 编号。按原子考点分组，每道答案使用
`#### <question_id>｜<题型>` 标题；选择题列“答案、解析、来源”，判断题列“结论、订正、来源”，
填空题列“答案、完整表述、来源”，简答题使用项目符号，计算题使用纵向计算过程。

教师题型蓝图只作为 `题目.md` 的题型配比参考，**不覆盖每原子考点至少 3 题规则**。
合同必须锁定原子考点题量、题型配比和总分参考值；仅创建用户明确要求的产物。

语言类改走 `language-accumulation-v1`：

- 查询单词或词组时，词性使用 `n/v/adj/adv/prep/phr v` 等缩写；含义使用英文释义，
  每个含义对应一个例句，并给出搭配与同义词/近义词，同时标明
  “已在词汇表/未在词汇表”；随后只问是否加入词汇语法积累表。
- 英译中先按最长词组优先拆分，标出表内/表外词条并解释，再给出自然译文。
- 中译英先拆中文语义单元，再映射为英文单词或词组并组合译文。
- 翻译表按规范化原句使用纯 Markdown `## 原句：...` 分组，原句、拆分和译文不得重复；每个关键表达
  子行只允许一个对应说明。两种翻译完成后逐项列出表外关键表达并询问
  是否加入词汇语法积累表；用户确认后才写入。翻译原句仅在用户要求保存时进入
  翻译积累表。
- 写作只积累英文句子。先展示中文含义、句型结构和可替换部分，再询问
  是否加入写作积累表。
- 三个积累表不使用 ID；词条、原句与英文句子本身分别作为去重和更新键。词汇表使用纯
  Markdown `## 词条：...` 分组，词条标题不得重复；每个义项占一个表格行并绑定一个搭配和例句。
  翻译按规范化原句去重，写作按规范化英文句子去重，所有已有主项只能增补而不能重复创建。
- 语言合同设置 `minimum_questions_per_point: 0` 和 `accumulation_only: true`，不执行题库校验。
- 语言查询前运行 `scripts/language_accumulator.py status` 区分已积累/未积累并返回跨表关联；
  用户确认后才调用 `add-vocab`、`add-translation` 或 `add-writing --confirmed`。脚本必须把新义项
  追加到原分组、拦截重复主项，并重建内部 `language_index.json`。
- 展示同义词/近义词时，比较核心语义、语气强弱和当前语境可替换性；辨析用于回答，不新增
  冗长表格列。
