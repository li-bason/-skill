## 二、Skill 在小艺平台的运行限制

### 2.1 运行环境

| 项目 | 内容 |
|------|------|
| 操作系统 | Linux |
| Shell | bash / sh（无 dpkg/apt） |
| Python | Python 3.10+（有 matplotlib、Pillow 等常用库） |
| 无预装 | ❌ LaTeX（pdflatex/xelatex） ❌ Pandoc ❌ wkhtmltopdf |
| 可安装 | 可通过 pip install 部分库（需用户确认） |
| 网络 | 可进行 WebSearch、WebFetch |
| 文件系统 | 仅限工作区沙箱内读写 |

### 2.2 运行时 Agent 拥有的工具（skill body 中可以调用）

Skill 被触发后，Agent 运行时会拥有以下全套工具：

| 工具分类 | 工具名称 | 用途 |
|---------|---------|------|
| **文件操作** | `Read` | 读取文件内容 |
| | `Write` | 写入/创建文件 |
| | `Edit` | 编辑文件（支持替换） |
| | `glob` | 查找文件路径 |
| | `grep` | 搜索文件内容 |
| | `list_files` | 列出目录 |
| **执行** | `bash` / `exec` | 执行 Shell 命令 |
| | `code` | 执行 Python/JS 代码 |
| **用户交互** | `ask_user_question` | 向用户提问并等待回答 |
| | `upload_file` | 上传文件给用户下载 |
| **信息获取** | `WebSearch` | 互联网搜索 |
| | `WebFetch` | 抓取网页内容 |
| **任务管理** | `todo_create/modify/list` | 任务进度跟踪 |
| **子代理** | `spawn_subagent` | 创建子 Agent 隔离执行任务 |
| | `task_tool` | 启动 skill_executor / grader 子代理 |
| **外部调用** | `invoke(funcName, params)` | 调用函数工具或 Agent 工具 |
| **技能管理** | `skill_tool` | 加载其他 Skill 的 SKILL.md |
| | `skill_complete` | 释放已加载的 Skill |

### 2.3 重要约束

1. **无 GUI/桌面** — Skill 只能生成文件产物，不能打开窗口或交互式界面
2. **文件路径** — 只能读写工作区沙箱内的文件，不能访问系统文件
3. **单轮交互** — SKILL.md body 是静态指令集，Agent 按指令一步步执行
4. **无持久服务** — Skill 被触发时启动一轮任务，完成后即结束，Shell 状态不跨轮保留
5. **无管理员权限** — 不能安装系统级软件包（如 TeX Live 等）
6. **Shell 隔离** — 每条 bash 命令之间，变量/函数/alias 不保留
7. **文件大小限制** — 单个文件过大时需分多次 Read（offset + limit）

## 三、依赖声明方式

在 SKILL.md frontmatter 的 `metadata` 中声明依赖：

```yaml
metadata:
  tools:
    - bundleName: "bundle_001"
      toolName: "weather_query"
  agents:
    - agentId: "aaabbbccc"
  clis:
    - name: "ohos-bm install"
```

### 三种依赖在 body 中的调用方式

| 依赖类型 | 调用语法 | 说明 |
|---------|---------|------|
| **函数工具** | `invoke(funcName:"toolName", params:{bundleName:"...", ...})` | bundleName 必填，其余参数按 arguments JSON Schema 构造 |
| **Agent 工具** | `invoke(funcName:"agent_as_a_tool", params:{agentId:"...", query:"..."})` | funcName 固定为 `agent_as_a_tool` |
| **CLI 工具** | 直接写命令，如 `ohos-bm install --module entry` | 通过 exec 或 bash 执行 |

## 四、可用工具/Agent/CLI

当前工作区：**没有注册任何外部工具、Agent 或 CLI**。

这意味着当前 `study-forge` Skill **不声明任何 tools/agents/clis**，完全依靠 Agent 内置能力（Read/Write/WebSearch 等）工作。

> **如果你在外部想添加工具调用**，需要：
> 1. 在 `<workspace>/resources/available-tools/` 下放置 `<bundleName>__<toolName>.json` 工具定义文件
> 2. 在 SKILL.md frontmatter 中声明 `metadata.tools`
> 3. 在 body 中用 `invoke(funcName, params)` 调用

## 五、SKILL.md 编写规范（核心红线）

### 5.1 Frontmatter 字段

```yaml
---
name: study-forge              # 必须：合法 ASCII kebab-case（小写字母、数字、连字符）
description: >-                # 必须：描述 skill 触发条件的文本
  为学生生成期末复习资料...      # 中文 ≤ 512 字符，英文 ≤ 1024 字符
metadata:                      # 可选：依赖声明
  tools: []
  agents: []
  clis: []
---
```

### 5.2 命名红线（不可违反）

- `name` 必须匹配目录名（如 `study-forge`）
- 必须合法 ASCII kebab-case：`^[a-z0-9-]+$`
- 不接受中文或其他非法格式作为 `name`
- 中文可用于 `description` 和正文

### 5.3 交付闸门

每次修改后必须通过 **gate 流水线**：

1. **validate** — 本地规范校验（检查结构、格式等）
2. **package** — 打包为 Skill 包
3. **upload** — 上传包（依赖 package 产物）
4. **safety_scan** — 安全扫描（不可跳过）

执行命令：`python3 -m scripts.gate <workspace>`

闸门为**非阻塞**：无论各阶段是否通过，结果都会反馈给用户，不阻止交付。

## 六、Skill Body 核心写作原则

### 6.1 目录结构规范

```
skill/<skill-name>/
├── SKILL.md
├── scripts/           # 辅助脚本
├── references/        # 参考资料
└── assets/            # 静态资源
```

所有路径使用**相对路径**（相对于 `skill/<skill-name>/`），不得硬编码绝对路径。

### 6.2 写作原则

- **描述用户意图，而非实现细节**：description 要写"什么场景触发"，而非"Read 文件然后 Write"
- **工具调用规范**：函数工具用 `invoke(funcName, params)`，CLI 用可执行命令字符串
- **步骤清晰**：用分阶段、分步骤的方式组织 body
- **每次只问一件事**：需要用户输入时，用 `ask_user_question` 依次提问
- **确认后再推进**：关键步骤后向用户确认

### 6.3 Tool Definitions 节

如果使用了外部工具，必须在 body 末尾添加工具定义节，格式如下：

**函数工具：**
```markdown
### Function: weather_query
- **toolName**: weather_query
- **description**: 查询指定城市的实时天气信息
- **参数**: {"type":"object","properties":{"city":{"type":"string","description":"城市名称"}},"required":["city"]}
- **约束**: （有顺序/幂等性要求时写）
- **语义**: （触发措辞路由时写）
```

**Agent 工具：**
```markdown
### Function: agent_as_a_tool
- **toolName**: aaabbbccc
- **description**: 查询出行相关资讯与方案
- **参数**: {"type":"object","properties":{"query":{"type":"string","description":"用户查询问题内容"}},"required":["query"]}
```

**CLI 工具：**
```markdown
### CLI: ohos-bm install
- **toolName**: ohos-bm install
- **description**: 安装应用包

| 调用 | 说明 | 样例 |
|------|------|------|
| `ohos-bm install --module <module>` | 安装指定模块 | `ohos-bm install --module entry` |
```

## 七、静态评估标准（skill-compass 会检查的维度）

如果你在外部写完 skill 后希望评估质量，skill-compass 评测会考察：

1. **SKILL.md 结构** — frontmatter 字段完整性、body 结构规范
2. **触发质量** — description 能否准确匹配用户意图
3. **安全性** — 是否有危险命令、硬编码凭据、路径穿越
4. **功能价值** — 解决问题的实际能力
5. **对比价值** — 相比无 skill 的基线提升
6. **规范合规** — 命名、路径、格式是否符合 spec

> **建议**：在外部完善时，重点关注以下内容：
> 1. **description** 的触发准确度 — 太窄则用户话术不命中，太宽则误触发
> 2. **脚本功能** — 可在 `scripts/` 下放 Python 脚本来辅助知识点提取、格式化输出等
> 3. **进度记录规范** — 每个阶段写 progress-log.md 是强要求
> 4. **出题策略** — 40% 基础 / 40% 中等 / 20% 综合的比例已在 SKILL.md 中固定
